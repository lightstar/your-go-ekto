# Паранормальный реестр - Агентство "Экто-Контроль"

Это учебный HTTP-сервис на Go для регистрации паранормальных сущностей, их досье и связанных с ними улик-изображений.

Проект выполнен в качестве решения финального задания курса по Golang от Павла Тарасова [Твой Golang - Продвинутый](https://stepik.org/course/284645/).

## Суть

Клиент отправляет досье в формате JSON и набор улик через `multipart/form-data`. Сервис проверяет входные данные, сохраняет валидные изображения в файловом хранилище и создает сущность с уникальным идентификатором UUID.

Если часть улик не прошла проверку, сервис сохраняет остальные и возвращает результат с частичным успехом и причинами отклонения каждого файла. Созданное досье и сохраненные изображения можно получить через HTTP API.

Проект демонстрирует:

- разделение HTTP, сервисного и инфраструктурного слоев;
- потоковую обработку multipart-запросов;
- ограничение размеров запросов и файлов;
- валидацию JSON и MIME-типа изображений;
- атомарную запись файлов через временные файлы;
- откат частично сохраненных данных при ошибках;
- классификацию и обработку ошибок;
- graceful shutdown HTTP-сервера;
- запуск напрямую и через Docker Compose.

## Возможности

- Создание досье с одной или несколькими уликами в форматах JPEG/PNG.
- Частичное сохранение, если некоторые улики невалидны.
- Получение досье по UUID.
- Получение сохраненных изображений с поддержкой HTTP Range.
- Файловое хранилище без внешней базы данных.
- Демонстрационный веб-интерфейс при запуске через Docker Compose.

## Основные ограничения

- до 10 улик в одном запросе;
- до 3 MB на одну улику;
- до 1 MB на досье;
- до 20 MB на все тело HTTP-запроса;

## Требования

Для запуска без Docker потребуется:

- Go 1.26.

Для контейнерного запуска потребуются:

- Docker;
- Docker Compose v2.

`make` является необязательным. Текущий `Makefile` рассчитан на Unix-подобное окружение с `/usr/bin/find`. В Windows его удобнее использовать через WSL или Git Bash; иначе можно выполнить соответствующие команды вручную.

## Запуск с Docker и Make

Собрать образ приложения:

```shell
make image
```

Запустить сервис, Nginx и демонстрационный интерфейс:

```shell
make run-in-docker
```

После запуска веб-интерфейс и API будут доступны по адресу:

```text
http://localhost:8080
```

Остановить контейнеры:

```shell
make stop-in-docker
```

Данные сохраняются в Docker volume `ekto-storage` и не удаляются обычной командой остановки.

## Запуск с Docker без Make

Собрать образ backend-сервиса:

```shell
docker compose -p your-go-ekto -f deployments/compose.yaml build your-go-ekto
```

Запустить все контейнеры:

```shell
docker compose -p your-go-ekto -f deployments/compose.yaml up -d
```

Веб-интерфейс и API будут доступны по адресу:

```text
http://localhost:8080
```

Посмотреть состояние контейнеров:

```shell
docker compose -p your-go-ekto -f deployments/compose.yaml ps
```

Остановить контейнеры:

```shell
docker compose -p your-go-ekto -f deployments/compose.yaml down
```

## Запуск без Docker с Make

Для запуска из исходного кода:

```shell
make run
```

По умолчанию сервер будет доступен по адресу:

```text
http://localhost:8080
```

Проверить его состояние можно запросом:

```shell
curl http://localhost:8080/health
```

Также приложение можно предварительно собрать:

```shell
make build
```

Результат сборки будет записан в `bin/ekto-server`.

Запуск собранного файла:

```shell
./bin/ekto-server
```

При нативном запуске поднимается только API. Демонстрационная страница из директории `web` работать не будет из-за CORS-ограничений.

## Запуск без Docker и Make

Запустить сервис непосредственно из исходного кода:

```shell
go run ./cmd/ekto-server
```

Либо собрать исполняемый файл:

```shell
go build -o bin/ekto-server ./cmd/ekto-server
./bin/ekto-server
```

По умолчанию данные сохраняются в директории `storage` относительно текущего рабочего каталога.

## Настройка запуска

Сервер поддерживает следующие аргументы командной строки:

| Аргумент   | Значение по умолчанию | Описание                                                 |
| ---------- | --------------------: | -------------------------------------------------------- |
| `-address` |               `:8080` | Адрес HTTP-сервера                                       |
| `-api`     |            `/api/v1/` | Префикс маршрутов API                                    |
| `-storage` |             `storage` | Путь к файловому хранилищу                               |
| `-log`     |               `debug` | Уровень логирования: `debug`, `info`, `warn` или `error` |

Пример:

```shell
go run ./cmd/ekto-server \
  -address=:9000 \
  -api=/api/v1/ \
  -storage=./data \
  -log=info
```

## HTTP API

| Метод  | Маршрут                       | Назначение                          |
| ------ | ----------------------------- | ----------------------------------- |
| `POST` | `/api/v1/entities`            | Создать сущность из досье и улик    |
| `GET`  | `/api/v1/entities/{id}`       | Получить досье по UUID              |
| `GET`  | `/api/v1/evidence/{filename}` | Получить сохраненную улику          |
| `GET`  | `/health`                     | Проверить состояние backend-сервера |

Запрос создания сущности должен иметь тип `multipart/form-data` и содержать:

- поле `dossier` с JSON-досье;
- одно или несколько полей `evidence` с изображениями.

Пример досье:

```json
{
  "name": "Призрак библиотеки",
  "description": "Раскидывает книги по ночам",
  "threat_level": 4,
  "vulnerabilities": ["соль", "громкая музыка"]
}
```
